# Survey system v2 — multi-survey + groups + targeting (2026-07-10)

Danny's call. Replaces the single global survey (migration 014) with multiple
surveys, optional grouping, and per-survey targeting (audience + placement).
Spec source: HANDOFF.md "NEXT UP — Survey system v2".

## Data model — migration 025_survey_v2.sql

- `survey_groups`: id, name, created_at.
- `surveys`: id, name, group_id (nullable fk, on delete set null), active
  (default false), placement, audience, sort_order, created_at.
  - placement check: `full_report_end | initial_report_end | account | post_purchase`
  - audience check: `all | first_report | first_purchase | promo_users | repeat_users`
- `survey_questions` gains `survey_id` fk (not null after backfill, on delete cascade).
- `survey_responses` gains `survey_id` fk (not null after backfill, on delete cascade).
- Backfill: create a default **"Launch survey"** (placement full_report_end,
  audience all, fixed uuid literal), attach every existing question/response
  to it, seed its `active` from the old `app_settings.'survey'.enabled` flag
  (coalesce false). The app_settings key is retired (left in place, unread).
- RLS: `surveys` public select where active; `survey_questions` select policy
  tightened to `active AND parent survey active`; response policies unchanged.

## Eligibility (src/lib/survey.ts)

Pure (unit-tested):
- `audienceMatches(audience, signals)` — signals = { completedReports,
  promoReports, completedPurchases } counts.
  - all → true; first_report → completedReports === 1; repeat_users → >= 2;
    promo_users → promoReports >= 1; first_purchase → completedPurchases === 1.
- `pickEligibleSurvey(surveys, answeredSurveyIds, signals)` — first active
  survey (sort_order, then created_at) the user hasn't answered whose
  audience matches. A user answers a given survey once.

I/O wrappers:
- `getAudienceSignals(service, userId)` — three count queries.
- `pickSurveyFor(service, requestClient, userId, placement)` — replaces
  `getSurveyCardData`; returns { show, surveyId, questions }. Missing-table
  (pre-migration) degrades to show:false, per project convention.
- The "already answered → thanks card" state goes away: an answered survey is
  simply never picked again (client keeps its in-session submitted state).

## Submission

- POST /api/survey body gains `survey_id`. Validate: survey exists + active
  (RLS-readable), active questions scoped to that survey, user hasn't
  answered that survey yet; insert rows with survey_id.

## Admin API

- `GET/POST /api/admin/surveys` — list (surveys + groups + per-survey question
  and respondent counts) / create survey.
- `PATCH/DELETE /api/admin/surveys/[id]` — survey update (name, group_id,
  active, placement, audience) / delete only when zero responses (else
  deactivate), cascade removes questions. **Question routes move** to
  `/api/admin/surveys/[id]/questions` (POST) and
  `/api/admin/surveys/[id]/questions/[qid]` (PATCH/DELETE) — same
  zero-responses delete rule as v1.
- `GET/POST /api/admin/surveys/groups`, `PATCH/DELETE .../groups/[id]`.
- `GET /api/admin/surveys/responses?survey=<id>|group=<id>` — per-survey
  aggregation (v1 shape) or group rollup.
- `POST /api/admin/surveys/summary` takes `{ survey_id }`.
- nav-status `surveyActive` = any active survey row (missing table → false).

## Admin UI

- AdminShell: `NavItem` gains `children` — Surveys becomes an expandable item
  with **Create survey** (`/app/admin/surveys`) and **Analytics**
  (`/app/admin/surveys/analytics`). Active pill on children; parent
  auto-expands + highlights when a child route is active; collapsed sidebar
  shows the parent icon linking to the first child.
- `/app/admin/surveys` — surveys grouped by group (+ Ungrouped), create/edit
  survey modal (modals-over-navigation rule), per-survey expandable question
  manager (v1 manager scoped by survey), active toggle per survey, group
  create/rename/delete.
- `/app/admin/surveys/analytics` — survey/group picker, v1 per-question
  aggregate rendering, AI summary scoped to the selected survey.

## Placements wired

- Report page: `report-page-content` resolves placement server-side —
  `full_report_end` when sections are populated (hasFullSections logic),
  else `initial_report_end`.
- Account page (`/app/account`): renders the `account` placement below ideas.
- `post_purchase`: enum + admin option exist, renders nowhere until payments
  (flagged in the admin UI copy).
