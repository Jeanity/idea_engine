# Phase 03 — Guided Follow-up Questions

## Goal ("done" looks like)
After confirming an idea, the user walks through a short archetype-specific question wizard (5–9 questions) that captures exactly what the research/report pipeline needs: business model shape, location specifics, resources on hand, and (for product ideas) rough cost inputs. Answers persist; users can leave and resume. This is the pitch's "asks the right questions" differentiator.

## Dependencies
Phase 2 complete (ideas exist with confirmed archetypes).

## Tasks (in order)

### 3.1 Design the question system
**Model: Opus** — prompt/schema design decision with a rebuild-trap risk.
Decide and document the hybrid approach: a **static base question bank per archetype** (predictable, cheap, testable) + **up to 3 dynamic follow-ups** generated by Claude from the idea text (the "smart" feel). Define the question JSON schema: `{key, text, input_type: text|select|number|multiselect, options?, required, maps_to}` where `maps_to` names the report-pipeline input it feeds (Phase 4 contract). Output: `docs/QUESTIONS.md` including the pet-treats worked example from the pitch (home-based? which pets? recipes/equipment? sales channel?).

### 3.2 Write the static question banks
**Model: Sonnet** — content quality matters (these ARE the product's perceived intelligence), not mechanical enough for Haiku.
One JSON file per archetype in `lib/questions/`, following the 3.1 schema. Product archetypes must capture cost-engine inputs: materials, equipment + wattage, active minutes per batch, batch yield, target hourly rate, sales channel.

### 3.3 Dynamic follow-up generation
**Model: Opus** for the prompt design (one prompt file); **Sonnet** for the API route.
Prompt takes idea text + archetype + static answers so far → returns 0–3 extra questions in the same JSON schema. Hard cap at 3; if generation fails, silently skip (static bank alone is a complete flow).

### 3.4 Wizard UI
**Model: Sonnet** — the main implementation lift of this phase.
One-question-per-screen wizard at `/app/ideas/[id]/questions`: progress bar, back navigation, input components per `input_type`, autosave each answer to `answers` table, resume from last unanswered question. On completion, set idea status to `researching`-ready and route to a "generate my report" screen (stub until Phase 4).

### 3.5 Answer summary screen
**Model: Haiku** — straightforward render of stored data.
Read-only recap of all answers with an "edit" link back into the wizard before generating the report.

### 3.6 Wizard tests
**Model: Haiku** — repetitive, low-ambiguity.
Tests: autosave persists, resume lands on correct question, required-field validation, dynamic-question failure degrades gracefully.

## Acceptance criteria
- [ ] Pet-treats idea produces questions covering: home-based vs stall vs online, pet type, equipment owned, batch inputs (yield, active time), sales channel — matching `docs/QUESTIONS.md`.
- [ ] Closing the tab mid-wizard and returning resumes at the right question with prior answers intact.
- [ ] Dynamic follow-ups appear for a rich idea and are absent (without error) when generation is disabled via env flag.
- [ ] Every question's `maps_to` key is documented — Phase 4 can be built against `docs/QUESTIONS.md` alone.
- [ ] Full flow works on mobile width (390px).

## Solo-operator sizing
~1–1.5 weeks part-time. The wizard UI is the bulk; keep it plain — no animations.
